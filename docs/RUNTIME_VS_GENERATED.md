# Runtime Vs Generated

This document explains the difference between Polystate runtime mode and generated mode, and what Polystate simplifies compared with using React Context, Redux, NgRx, and Angular Signals directly.

## The Two Ways To Use Polystate

Polystate supports two distinct approaches.

### Runtime Mode

In runtime mode, you use the Polystate store directly at application runtime.

Typical packages:

1. `@polystate/core`
2. `@polystate/react`
3. `@polystate/angular`

Typical flow:

1. Create a store with `createStore(...)`.
2. Define actions as pure functions.
3. Bind that store to React with hooks or to Angular with a service.

Example idea:

```ts
const store = createStore(initialState, actions);
```

Then:

1. React uses `useStore`, `useSelector`, `useDispatch`.
2. Angular uses `PolystateService`, `select`, `select$`, or `createAngularService`.

### Generated Mode

In generated mode, you define your state once in a framework-agnostic definition file, then generate framework-native code.

Typical packages:

1. `@polystate/definition`
2. `@polystate/cli`
3. `@polystate/generator-react`
4. `@polystate/generator-angular`

Typical flow:

1. Write `store.definition.ts`.
2. Run `polystate generate`.
3. Use the generated Redux or NgRx files in your application.

Example idea:

```bash
npx polystate generate store.definition.ts --react --angular
```

Then:

1. React gets generated Redux store, actions, selectors, and hooks.
2. Angular gets generated NgRx state, actions, reducer, selectors, facade, and module.

## Simple Difference

The shortest way to think about it is this:

1. Runtime mode means Polystate is your state management runtime.
2. Generated mode means Polystate is your state management authoring tool.

That is the key distinction.

In runtime mode, your app runs on Polystate primitives.
In generated mode, your app runs on Redux or NgRx output generated by Polystate.

## Runtime Mode In Practice

Runtime mode is usually better when:

1. You want the fastest setup.
2. You are building one React app or one Angular app.
3. You do not need Redux or NgRx as an organization standard.
4. You want to keep state logic close to the feature.

Runtime mode gives you:

1. One store definition directly in code.
2. Simple actions as immutable state transforms.
3. Selective subscriptions.
4. Middleware support.
5. A smaller cognitive surface than full Redux or NgRx.

Tradeoffs:

1. You are using Polystate-specific runtime APIs.
2. If a team requires native Redux or NgRx structure, runtime mode is not the target shape.

## Generated Mode In Practice

Generated mode is usually better when:

1. Your React team wants Redux Toolkit output.
2. Your Angular team wants NgRx output.
3. You want one shared store definition used by multiple frameworks.
4. You want framework-native generated files instead of hand-written boilerplate.

Generated mode gives you:

1. One framework-agnostic source of truth.
2. Generated Redux for React.
3. Generated NgRx for Angular.
4. Less repeated action and selector setup.
5. Cleaner cross-framework state modeling.

Tradeoffs:

1. You now have a generation step.
2. Generated files are an artifact you should treat as output, not hand-maintained source.
3. Complex typing and advanced generation patterns depend on the maturity of the generator.

## Runtime Vs Generated Side By Side

| Topic | Runtime Mode | Generated Mode |
|---|---|---|
| Primary goal | Simpler runtime state management | Framework-native code from one definition |
| What you write | Store and actions directly | Store definition file |
| What the app runs on | Polystate store | Redux or NgRx generated code |
| Best for | Single app, fast development | Multi-framework teams, native store ecosystems |
| React integration | `@polystate/react` hooks | Generated Redux store and hooks |
| Angular integration | `@polystate/angular` service and selectors | Generated NgRx store and facade |
| Boilerplate level | Low | Low in authored source, generated output can be verbose |
| Shared cross-framework state | Possible, but manual | Native strength of the approach |

## What Polystate Simplifies In React

### Compared With React Context Alone

Plain React Context is often used as a simple state transport layer, but it does not give you a real state-management model by itself.

Without Polystate, teams often end up writing:

1. Context creation.
2. Provider wiring.
3. State mutation functions.
4. Derived selectors manually.
5. Re-render management manually.

What Polystate simplifies:

1. Store creation becomes explicit and reusable.
2. Actions become named immutable transitions instead of ad hoc callbacks.
3. `useSelector` gives slice-based subscriptions instead of broad context re-renders.
4. Middleware gives logging, persistence, and devtools integration without custom plumbing.
5. The store logic can live outside component trees.

In short:

React Context answers how to pass state.
Polystate answers how to model, update, subscribe to, and reuse state.

### Compared With Writing Redux By Hand

Plain Redux or Redux Toolkit gives you a strong architecture, but there is still repetitive setup.

Without Polystate, teams often write:

1. Slice files.
2. Action wiring.
3. Selectors.
4. Typed hooks.
5. Boilerplate around shared domain logic.

What Polystate simplifies in generated React mode:

1. One `store.definition.ts` replaces repeated slice authoring.
2. Hooks are generated.
3. Selectors are generated.
4. Middleware wiring is included in generated store output.
5. A shared state model can also generate Angular output from the same source.

What Polystate simplifies in runtime React mode:

1. You avoid Redux setup entirely.
2. You still get named actions and selective subscriptions.
3. You keep the model simple for smaller apps.

## What Polystate Simplifies In Angular

### Compared With Angular Signals Alone

Angular Signals are a great primitive, but on their own they do not define an opinionated store model.

Without Polystate, teams often write:

1. Multiple `signal(...)` declarations.
2. Manual update methods.
3. Derived `computed(...)` wiring.
4. Cross-feature orchestration manually.
5. Observable bridges manually if templates or services still rely on RxJS.

What Polystate simplifies:

1. One store becomes the source of truth instead of many loosely related signals.
2. Actions define state transitions consistently.
3. `select(...)` and `select$(...)` expose the same store to both Signals and Observables.
4. The store model remains reusable and testable.
5. Angular services become thinner because state updates go through named actions.

In short:

Signals are a primitive.
Polystate adds a structured store model on top of those primitives.

### Compared With Writing NgRx By Hand

NgRx is powerful, but it comes with a known setup cost.

Without Polystate, teams often write:

1. State interfaces.
2. Actions.
3. Reducers.
4. Selectors.
5. Facades or store wrappers.
6. Feature module registration.

What Polystate simplifies in generated Angular mode:

1. One store definition generates NgRx actions, reducers, selectors, facade, and module.
2. Components can consume a facade instead of raw store plumbing.
3. The same business state model can also generate React output.
4. Teams reduce duplicate work across frameworks.

What Polystate simplifies in runtime Angular mode:

1. You avoid full NgRx setup when it is too heavy for the problem.
2. You keep a service-based Angular API.
3. You still get structured actions and selectors.
4. You can expose Signals and Observables from one place.

## How To Choose

Choose runtime mode if:

1. You want simple state management without Redux or NgRx ceremony.
2. You are working in a single framework.
3. Your team prefers a smaller abstraction surface.

Choose generated mode if:

1. Your team already standardizes on Redux Toolkit or NgRx.
2. You want framework-native output for integration with existing app architecture.
3. You want to share domain state definitions across React and Angular.

## What The Package Really Saves You From

Polystate mainly reduces four kinds of repeated work.

### 1. Repeated Store Wiring

Without Polystate:

1. React Context providers are manually wired.
2. Redux slices and typed hooks are manually wired.
3. Angular services or NgRx facades are manually wired.

With Polystate:

1. One store model drives framework integration.
2. The integration layer is thinner and more predictable.

### 2. Repeated Action Definitions Across Frameworks

Without Polystate:

1. React app writes one set of state transitions.
2. Angular app writes the same logic again in another style.

With Polystate:

1. One action model can serve both.
2. Generated mode is especially strong here.

### 3. Repeated Selector And Subscription Setup

Without Polystate:

1. You manually define derived access patterns.
2. You manually keep them aligned across frameworks.

With Polystate:

1. Runtime mode gives direct selector-based subscription APIs.
2. Generated mode gives generated selector output and facade usage.

### 4. Repeated Boilerplate Around State Tools

Without Polystate:

1. Redux requires slice and hook setup.
2. NgRx requires action, reducer, selector, and facade setup.
3. Context often grows ad hoc and loses structure.
4. Signals alone do not solve shared action patterns.

With Polystate:

1. State changes are modeled once.
2. Framework integration becomes secondary instead of driving the architecture.

## Short Recommendation Matrix

### Use Runtime Mode When

1. Your app is small to medium.
2. You want simple structured state.
3. You do not need Redux or NgRx as a formal requirement.
4. You want React hooks or Angular service ergonomics directly.

### Use Generated Mode When

1. Your organization already uses Redux Toolkit or NgRx.
2. You need a framework-native architecture.
3. You want one domain definition shared across apps or teams.
4. You want to minimize handwritten store boilerplate.

## Final Summary

Polystate is not just another store library.

Its real value is separating domain state design from framework-specific state plumbing.

Runtime mode simplifies state management by replacing custom Context patterns, manual Signal orchestration, and lightweight ad hoc stores with one structured store model.

Generated mode simplifies framework-native state management by replacing repeated Redux and NgRx boilerplate with one shared source definition.

If you want the shortest summary:

1. Runtime mode simplifies how you run state.
2. Generated mode simplifies how you author state for multiple frameworks.
